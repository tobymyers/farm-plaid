The Business Opportunity

Over the past months, I've had lots of conversations with farmers and agtech companies, and a theme has emerged; there's a lot of pain around getting and managing field-level data. On the farmer side, most apps demand similar information: field drawings, crop details, and harvest timelines. A farmer that works with 3 Agtech SaaS platforms has to manage and update the same information in 3 places. Farmers don't have time for that.

On the tech side, I've heard from leaders in the space - large incumbents and unicorns alike - how difficult it can be to get farmers to input data into their platforms and maintain it once it's there. Part of the problem is likely that farmers already have to manage data in so many places - there's little appetite for another app asking for the same information again. In many cases, the value that these companies can deliver to the their farmer customers (or other supply chain actors) is directly predicated on having quality field level data, much of which needs to be input manually. Lack of access to this data is a big, expensive problem.

The Product Management Opportunity

I'm a big advocate for falling in love with the problem, rather than any individual solution to it. I'm not sure if this particular idea - a kind of "Plaid for Farm Data" - is a good one. If I had to bet on the business model laid out on the landing page, I'd probably bet against it.

That being said, the value of this project is two-fold. First, think of the impact generative AI can have on speed of prototyping and testing in the design phase. As a pm, I'm always looking for ways to increase the odds that a development teams' hard work pays off. Anything that allows a product trio to move faster and test more easily with minimal investment is a win, and should lead to more validated product specs, better bets, and code that drives more customer and business value.

Second, this project brought me further into the technical details than product people usually get the chance to go. I started with a sqllite db and a Flask App running locally, and then hosted the app on Heroku with a postgres sql database. When I wanted to set up my local database on postgres sql as well for smoother testing and deployment, I had to face and overcome my own technical debt created hours before. There were moments of amazement, like when I asked ChatGPT to produce css matching the Airbnb style and it did so flawlessly. There were also many moments of frustration, like when I spent 3 hours working through SQLAlchemy / Python versioning issues as I tried to run a database migration and deploy to Heroku.

After 20 hours, I ended up with a functional development + production environment, a user facing landing page that could be used to validate farmer desirability assumptions, and a beta public API that could be used to test desirability on the agtech side.

The tools I used to build this include Leaflet.js for mapping, Dbeaver for db visualization, Heroku for hosting, iTerm2 because I got tired of the Mac Terminal, Python + Flask on the backend, Javascript + CSS + html on the front end without any particular framework, Postgressql for both the local and Heroku DB, and Visual Code Studio as an editor, SQL Alchemy as an ORM, and of course ChatGPT 3.5 for answering questions, editing code, interpreting console errors, etc

The process thus far has improved my technical fluency and prompt engineering skills, and reaffirmed my respect for the expertise of real developers and designers. Their work goes far beyond what a chatbot can achieve.
